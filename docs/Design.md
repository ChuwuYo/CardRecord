# 刷记架构与设计

本文只维护长期有效的 WHY、边界和不变量。具体版本号、依赖和命令以构建文件与 [`AGENTS.md`](../AGENTS.md) 为准。

## 1. 产品边界

刷记用于手动记录卡片消费次数，并按年费统计窗口显示进度。它不连接银行、不做记账、没有账号或后端。

数据保存在设备本地：Room 存卡片、文件夹与流水，DataStore 存外观设置。用户可通过系统文件选择器主动导入/导出 JSON；系统云备份和设备迁移关闭。

## 2. 架构

```text
Compose Screen
    ↓ callbacks / StateFlow
ViewModel
    ↓ use cases
Repository
    ↓
Room DAO / DataStore / SAF
```

- UI 使用单向数据流；业务状态由 ViewModel 持有，Composable 只渲染和发送意图。
- Repository 收口跨表事务、日期规则、导入导出和 DAO 访问。
- `AppContainer` 负责轻量依赖装配和跨页面事件；当前 Repository 仍向 ViewModel 暴露部分 Room Entity / 投影，不宣称完整领域隔离。
- 同一概念只有一个真源：卡片底色来自 `colorArgb`，周期来自 `AnnualFeeCycle`，卡组织视觉布局来自共享卡面布局函数。
- 跨页面一次性结果进入有界事件队列：短暂没有订阅者时不丢失，消费后不因界面重建而重放。

## 3. 数据模型

Room v8 有三张表：

- `cards`：卡片文本、三态卡类型、信用卡账单日/还款日、有效期、下次结算日、目标笔数、底色、卡面样式、卡组织、朝向、文件夹和创建时间。
- `transactions`：流水主键、卡片主键和发生时间；删除卡片时 `CASCADE`。
- `card_folders`：名称、颜色、排序和创建时间；删除文件夹时卡片的 `folder_id` 通过 `SET NULL` 归为未分类。

`currentCount` 不存表。Repository 合并卡片与流水，在 Kotlin 中按当前周期派生 `CardWithCount`，避免“存储计数”和“流水事实”成为两个会漂移的真源。

卡类型使用稳定 key `UNSPECIFIED / DEBIT / CREDIT`。历史行和 schema 1 备份没有这项事实，迁移后保持 `UNSPECIFIED`，不根据名称、卡组织或年费设置猜测。账单日与还款日是可空的每月日号（1..31）；只有 `CREDIT` 可以持久化这两个值，其他类型在写入边界清空。

### 写入不变量

- 禁止用 SQLite `REPLACE` 更新被外键引用的父表。
- 新卡走 `@Upsert`；编辑走 `@Update` 并检查受影响行数。编辑期间卡片若已删除，保存失败且不会重新插入。
- 删除卡片必须先二次确认，确认后永久删除卡片及其流水。
- 文件夹计数由单条响应式查询返回，不做逐文件夹 N+1 查询或手动刷新。

## 4. 日期与年费周期

`validUntilMillis` 与 `nextDueDateMillis` 是 UTC 零时编码的日历日期令牌，不是直接比较的绝对瞬间。

- 有效期：按用户当前时区的本地日期判断；截止日当天仍有效，进入次日才过期。
- 年费窗口：`[上一年同月日, 下次结算日)`，按日历年计算，不使用固定 365 天。
- 二月月末：年费日期的 2 月 28/29 日都按对应年份的二月末处理；普通日期编码不附带这条业务归一化。

`AnnualFeeCycle` 用 sealed 状态表达不可变事实：

- `Unscheduled`：未设置结算日，兼容旧行为并统计全部流水。
- `Upcoming`：统计窗口未开始；不显示误导进度，也不允许记一笔。
- `Active`：只接纳窗口内流水。
- `Overdue`：等待结算日归一化，暂不写入。

过期结算日按年份差直接推进到未来，复杂度为 O(1)，只修改日期，不删除历史流水。应用在前台边界重新计算日期与时区，避免长时间打开后状态过期。

## 5. 卡组织与卡面

卡组织和卡面来源是两个独立维度：

- `imageProviderKey`：可空的卡组织稳定 key。
- `imageSourceType`：`NONE` 纯色、`PROVIDER` 预设、`USER` 用户图片。

共同规则：

- `colorArgb` 始终是底色唯一来源，卡组织不能覆盖用户选色。
- `NONE` 与 `PROVIDER` 可显示右下品牌徽标；`PROVIDER` 额外显示右上水印和相交光环。
- `USER` 原样显示用户图片，只叠加文字，不显示徽标、水印、斜条或光环。
- 用户图片模式隐藏卡组织选择器但保留已存 key；切回纯色或预设后恢复选择。
- 横版、竖版和双列紧凑布局显式选择布局模式。紧凑布局复用同一装饰组的统一缩放与锚点，并为文字、右上装饰和右下徽标分别保留安全区。
- 卡面文字与图标使用同形暗色投影保证浅色背景可读；半透明徽标底板、水印和光环保留原透明度，并用同形暗色底层补足轮廓，不改写品牌色。
- 未知持久化 key 由集中解析函数安全回退；不能在调用点直接 `valueOf`。
- 用户卡面图片的持久化读取授权由单一组件管理。数据库引用与编辑中的临时选择共同决定授权是否保留；保存、放弃、删除、导入和进程启动后统一回收无引用授权，避免泄漏或误删共享 URI。

## 6. UI 状态与失败语义

- 列表的卡类型与文件夹筛选共享一个明确的筛选模型；文件夹删除后不会保留幽灵筛选。筛选底部面板用单一 Lazy 列表承载任意数量的文件夹，并把管理入口固定在滚动区外；Lazy 项使用稳定 ID/key，不用可重复标题作 key。
- 详情页区分 `Loading / Loaded / Missing`。删除成功事件发出后才导航返回；失败保留当前页面。
- 编辑页区分 `New / Loading / Ready / Missing / Failed`。加载失败可重试，目标不存在时不显示空白“新建卡”表单。
- 流水历史是 LazyColumn 的真实 item，不把全部历史一次性组合在单个 item 中。
- 主题加载完成后再建立 Compose 主题；同一深浅色决策同时驱动内容、系统栏与 AppCompat 资源限定符。DataStore 是设置真源，同步启动缓存只负责在 Activity 创建前恢复明暗方向。

## 7. 备份与恢复

备份当前导出 schema `2`，顶层对象显式包含版本号与三张表；导入同时支持 schema `1`。schema 1 卡片缺少类型、账单日与还款日，导入时明确补为 `UNSPECIFIED / null / null`。备份 DTO 独立于 Room Entity，数据库实现细节变化不能无意改变公开文件格式；后续格式变更必须提升版本并保留明确的兼容边界。

### 导出

- 三张表在同一个 Room 只读事务快照中读取，避免并发写入产生不一致文件。
- 同一快照先编码到不保留内容的限流计数 sink；只有输出可被本版本重新导入时才打开 SAF 目标，超限不会破坏已有目标。
- 实际 JSON 直接编码到输出流，不额外构造完整字符串；截断写入避免较短备份残留旧文件尾部。开始写 Provider 后的平台 I/O 故障仍可能留下部分文件，单个 document URI 不具备通用原子替换能力。

### 预览与导入

- 预览和正式导入复用同一套受限流式解码、版本检查和结构校验。
- 预览记录完整文件内容的 SHA-256；用户确认后导入必须重新读取并匹配该摘要，防止确认期间文件被替换。
- 输入流有固定大小上限；重复或非正主键、非正目标笔数和缺失必填时间字段在写库前拒绝。
- `REPLACE` 与 `MERGE` 都把备份 ID 仅视为文件内部引用标签，写入时分配新 ID，并用本次备份内的映射重写外键，避免碰撞现库或消耗异常大的自增空间。
- `REPLACE` 在一个事务内清空旧数据，再按 folder → card → transaction 写入；为保证完整恢复，备份内存在孤立流水时拒绝整个文件并回滚。`MERGE` 保留现有数据；备份未包含的 folder 不能因为数值恰好相同而绑定到现库文件夹，该引用归为未分类；孤立流水跳过并计数。
- 写入和过期结算日归一化属于同一个事务。读取、校验和事务执行阶段都可取消并回滚；全部写入完成、进入 SQLite 提交边界后不再接受显式取消，提交与图片授权收尾必须给出确定结果。
- inspect/export/import 互斥。可取消阶段会同时取消协程并关闭当前流，使阻塞 I/O 能退出；UI 在提交已开始时保持处理中并等待回执。
- 导入文件只使用系统选择器本次授予的读取 URI，不申请持久化权限；确认状态只在当前进程内保留，日志不记录完整 URI。

## 8. 数据库迁移

所有 v1→v8 迁移显式注册在 `AppDatabase.ALL_MIGRATIONS`，数据库打开采用 fail closed，没有 destructive fallback。v7→v8 只增加卡类型与两个信用卡日号字段，旧行统一保持未选择。

迁移遵循：

- 重建父表时先处理/重连子表外键，不能依赖事务内无效的 `PRAGMA foreign_keys=OFF`。
- 历史文件夹、卡片和流水必须保留；悬空 folder 引用可归一化为 `NULL`。
- 当前 schema 导出到 `app/schemas/`；任何列、约束或索引变化都要同步迁移、schema 与 `MigrationTest`。

## 9. 设置、国际化与主题

- DataStore 只把 `IOException` 当作可恢复读取故障；编程错误和其他异常继续暴露。
- ThemeMode / ColorSource 等持久化值使用显式 key 和集中未知值回退。
- 用户可见文本全部来自简中/英文资源；数量文案使用 plurals，自动测试校验 string/plurals 的 key、quantity 与格式化占位符一致。
- 全局主题种子色与单张卡 `colorArgb` 是不同概念，不能互相覆盖。

## 10. 验证与发布边界

- JVM 测试覆盖周期、日期、迁移、Repository、备份和 ViewModel 状态机；Robolectric + Room in-memory 验证 Android/数据库边界。
- `ktlint`、单测、Lint、构建、模拟器/真机和 GitHub Actions 是不同证据层级，报告时必须区分。
- 单一 CI 工作流对 push / PR / 手动运行执行格式、单测与 Debug/Release Lint；签名 job 通过 `needs` 依赖同一提交的验证结果。正式签名 Secrets 只注入验证通过且仍为当前头的 `main` push 或手动运行的签名步骤，发布写权限再隔离到后续 job；密钥材料不进入仓库或构建 artifact。仓库规则与受保护 Environment 属于 GitHub 侧权限边界，不能由应用代码替代。
- `apk-latest` 的运行在签名前、发布前和切换 tag 前后都会核对 `main` 当前头，旧提交的 Re-run 不得回退滚动包。发布先上传暂存资产，并在有界等待内核对 GitHub 返回的精确摘要，再切换固定资产、元数据与 tag；失败处理会恢复并核验旧状态。新状态与下载摘要全部通过后才进入清理阶段，成功运行还会收口中断遗留的暂存 APK，并断言只剩一个固定资产。
- 发版清单见 [`RELEASE_CHECKLIST.md`](RELEASE_CHECKLIST.md)，用户可见变化见 [`CHANGELOG.md`](CHANGELOG.md)。

## 11. 关键取舍

- 不存 `currentCount`：牺牲少量运行时聚合，换取流水单一真源。
- 手写 DI：项目依赖图小，避免引入更重的框架；依赖仍通过构造函数显式注入。
- 不提供不完整删除撤销：若未来支持撤销，必须同时恢复卡片与全部流水，或采用完整软删除。
- 不做后台周期任务：产品没有后台提醒需求；前台边界重算足以保持展示与写入正确。
- 不新增独立统计起点字段：统计起点可从下次结算日稳定派生，避免多一份会漂移的年度状态。
